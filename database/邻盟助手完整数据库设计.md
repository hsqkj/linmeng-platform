# 邻盟助手完整数据库设计文档

## 数据库概述

**数据库名称**: neighbor_alliance  
**数据库类型**: MySQL 8.0+  
**字符集**: utf8mb4  
**排序规则**: utf8mb4_unicode_ci  

## 设计原则

1. **用户角色分离**: 商家端和居民端用户分表管理
2. **数据规范化**: 遵循第三范式，减少数据冗余
3. **性能优化**: 合理设计索引，支持高并发查询
4. **扩展性**: 预留扩展字段，支持业务迭代
5. **安全性**: 敏感数据加密存储，支持数据脱敏

## 一、基础字典表

### 1.1 社区字典表 (communities)

**用途**: 存储社区基础信息，支持商家和居民的社区归属

```sql
CREATE TABLE IF NOT EXISTS communities (
    id INT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '社区ID',
    name VARCHAR(100) NOT NULL COMMENT '社区名称',
    city VARCHAR(50) NOT NULL COMMENT '所属城市',
    district VARCHAR(50) NOT NULL COMMENT '所属区县',
    street VARCHAR(100) DEFAULT NULL COMMENT '所属街道',
    address VARCHAR(255) DEFAULT NULL COMMENT '详细地址',
    latitude DECIMAL(10, 7) DEFAULT NULL COMMENT '纬度',
    longitude DECIMAL(10, 7) DEFAULT NULL COMMENT '经度',
    population INT DEFAULT 0 COMMENT '常住人口',
    household_count INT DEFAULT 0 COMMENT '户数',
    area DECIMAL(10, 2) DEFAULT 0.00 COMMENT '面积（平方公里）',
    contact_person VARCHAR(50) DEFAULT NULL COMMENT '联系人',
    contact_phone VARCHAR(20) DEFAULT NULL COMMENT '联系电话',
    description TEXT COMMENT '社区简介',
    status TINYINT DEFAULT 1 COMMENT '状态：0-禁用，1-正常',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (id),
    KEY idx_name (name),
    KEY idx_city_district (city, district),
    KEY idx_location (latitude, longitude),
    KEY idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='社区字典表';
```

## 二、商家端表结构

### 2.1 商家账号表 (merchant_accounts)

**用途**: 商家登录认证信息（账号与信息分离）

```sql
CREATE TABLE IF NOT EXISTS merchant_accounts (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '商家账号ID',
    openid VARCHAR(100) DEFAULT NULL COMMENT '微信OpenID',
    unionid VARCHAR(100) DEFAULT NULL COMMENT '微信UnionID',
    phone VARCHAR(20) NOT NULL COMMENT '手机号',
    password VARCHAR(255) DEFAULT NULL COMMENT '密码（bcrypt加密）',
    salt VARCHAR(32) DEFAULT NULL COMMENT '密码盐值',
    login_type TINYINT DEFAULT 1 COMMENT '登录方式：1-手机号，2-微信',
    last_login_time DATETIME DEFAULT NULL COMMENT '最后登录时间',
    last_login_ip VARCHAR(50) DEFAULT NULL COMMENT '最后登录IP',
    login_count INT DEFAULT 0 COMMENT '登录次数',
    status TINYINT DEFAULT 1 COMMENT '状态：0-禁用，1-正常，2-待审核',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_phone (phone),
    UNIQUE KEY uk_openid (openid),
    KEY idx_status (status),
    KEY idx_create_time (create_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='商家账号表';
```

```sql
    KEY idx_community_id (community_id),
    KEY idx_location (latitude, longitude),
    KEY idx_rating (rating),
    KEY idx_verify_status (verify_status),
    KEY idx_status (status),
    FOREIGN KEY (merchant_id) REFERENCES merchant_accounts(id) ON DELETE CASCADE,
    FOREIGN KEY (industry_id) REFERENCES industries(id) ON DELETE SET NULL,
    FOREIGN KEY (community_id) REFERENCES communities(id) ON DELETE SET NULL

```

## 十、数据库视图

### 10.1 商家活动统计视图

```sql
CREATE OR REPLACE VIEW v_merchant_activity_stats AS
SELECT 
    m.id AS merchant_id,
    m.name AS merchant_name,
    COUNT(DISTINCT a.id) AS total_activities,
    COUNT(DISTINCT CASE WHEN a.status = 1 THEN a.id END) AS published_activities,
    COUNT(DISTINCT CASE WHEN a.status = 0 THEN a.id END) AS draft_activities,
    SUM(a.view_count) AS total_views,
    SUM(a.like_count) AS total_likes,
    SUM(a.share_count) AS total_shares,
    SUM(a.comment_count) AS total_comments,
    SUM(a.order_count) AS total_orders,
    SUM(a.order_amount) AS total_amount
FROM merchant_accounts ma
INNER JOIN merchant_info m ON ma.id = m.merchant_id
LEFT JOIN marketing_activities a ON ma.id = a.merchant_id
GROUP BY m.id, m.name;
```

```sql
CREATE OR REPLACE VIEW v_activity_detail AS
```

### 10.3 用户订单统计视图

```sql
CREATE OR REPLACE VIEW v_user_order_stats AS
SELECT 
    u.id AS user_id,
    u.nickname,
    u.avatar,
    COUNT(DISTINCT o.id) AS total_orders,
    COUNT(DISTINCT CASE WHEN o.status = 0 THEN o.id END) AS pending_orders,
    COUNT(DISTINCT CASE WHEN o.status IN (1, 2) THEN o.id END) AS processing_orders,
    COUNT(DISTINCT CASE WHEN o.status = 3 THEN o.id END) AS completed_orders,
    SUM(o.pay_amount) AS total_amount
FROM resident_users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.nickname, u.avatar;
```

## 十三、数据库设计总结

### 13.1 表结构统计

| 模块 | 表数量 | 主要表 |
|------|--------|--------|
| 基础字典 | 3 | communities, industries, marketing_templates |
| 商家端 | 3 | merchant_accounts, merchant_info, merchant_memberships |
| 居民端 | 2 | resident_users, resident_addresses |
| 营销活动 | 3 | marketing_activities, activity_drafts, channel_contents |
| AI服务 | 2 | ai_optimization_logs, ai_configs |
| 用户交互 | 5 | user_collections, user_follows, user_likes, user_comments, user_consultations |
| 订单管理 | 2 | orders, order_reviews |
| 数据统计 | 3 | activity_statistics, merchant_statistics, user_behavior_logs |
| 系统管理 | 4 | system_configs, operation_logs, notifications, files |
| **总计** | **27** | |

### 13.2 设计亮点

1. **用户角色分离**: 商家和居民分表管理，便于权限控制和业务扩展
2. **账号信息分离**: 商家账号与店铺信息分离，支持多店铺场景
3. **多渠道支持**: channel_contents表支持不同平台内容定制
4. **AI服务记录**: 完整记录AI优化过程，支持成本核算和质量评估
5. **行为日志**: user_behavior_logs表支持用户行为分析和精准营销
6. **统计汇总**: activity_statistics和merchant_statistics支持快速数据查询
7. **全文搜索**: 支持活动内容和商家名称的全文检索
8. **扩展性强**: JSON字段支持灵活的业务扩展

### 13.3 性能优化

1. 合理设计索引，覆盖常用查询场景
2. 分表统计避免实时聚合查询
3. 日志表独立，不影响核心业务性能
4. 支持读写分离和分库分表扩展

### 13.4 安全性

1. 敏感字段加密存储（密码、手机号等）
2. 操作日志完整记录，支持审计追溯
3. 外键约束保证数据一致性
4. 软删除机制，数据可恢复
